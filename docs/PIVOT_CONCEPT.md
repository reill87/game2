# 판교 IT 회사 키우기 — 컨셉 디자인 v0.1

> 게임의 차별화를 한국 IT 회사 시뮬 메타로 옮기는 피벗 기획서.
> v1 메커닉 검증을 마친 시점(2026-05-08)에서 시작. 코드와 함께 다듬어 가는 살아있는 문서.

---

## 1. 한 줄 정의

**"분당 셰어오피스에서 시작해 강남 사옥까지 — 한국 IT 회사 시뮬"**

플레이어는 5인 미만 IT 스타트업의 대표. 매주 자동으로 흐르는 시간 속에서 프로젝트 런칭 → 매출 → 사옥 이전 → 채용을 반복하며 회사를 키운다. 그 사이사이 단톡·조직개편·장애콜·임원 PT 같은 한국 IT 특유 모먼트가 끼어들어 결정을 강요한다.

기존 v1의 "인디 게임 스튜디오" 메타에서 이쪽으로 피벗. 본가 콘텐츠(『게임개발 스토리』 류)가 점유하지 못한 영역.

## 2. 차별화 3축 — "이 게임의 맛"

| 축 | 무엇 | 왜 |
|---|---|---|
| **사무실 진행 narrative** | 분당 → 판교 → 강남, 단계마다 시각·인원·문화가 바뀜 | 보상 루프의 가시화. "성장하는 회사를 보는 재미" |
| **한국 IT 직격 이벤트 풀** | 36+ 후보 / 24+ 구현 | "이거 우리 회사잖아" 모먼트. 본가가 절대 못 들어옴 |
| **직원 페르소나 + 진짜 퇴사** | 트레이트 메커닉 + 사기 30↓ N주 → 실제 퇴사 | 사기 메트릭에 진짜 무게. 야근 ON 전략의 진짜 비용 |

추가 후보(선택):
- **트렌드 메타** — 매 분기 G/T modifier 변동 (AI 봄 / 메타버스 침체 / Web3 거품 같은 시대 풍자)
- **사옥 단계별 일러스트** — 단계 진행의 시각 보상

## 3. 사무실 단계 — Narrative Arc

성장이 한 곳에서 정체되지 않게, 사옥 자체를 보상 루프의 핵심 축으로. 단계마다 시각·인원·문화가 가시적으로 달라진다.

| 단계 | 이름 | 인원 | 비용 | 주요 변화 | 시각 큐 |
|---|---|---|---|---|---|
| 1 | 분당 셰어오피스 | 3명 | 시작 | PM·디자이너·개발자 3인. 단톡 1개. | 좁은 데스크, 노트북 1대씩 |
| 2 | 판교 임대 사무실 | 4명 | 300g | QA 합류. 캡슐 커피머신. 회의실 1개. | 트인 오픈 스페이스 |
| 3 | 판교 자체 사옥 | 5명 | 800g | DevOps/그로스 1명. 옥상·라운지. | 자체 빌딩 일부 층 |
| 4 (v2) | 강남 사옥 | 6명 | 2000g | 신규 사업·해외 거점. | 강남 오피스 빌딩 |

각 단계 진입 = "**판교 입성**" 같은 짧은 컷 연출 + 회사 단톡에 축하 이모지 폭탄. 게임의 클라이맥스 중 하나.

v1 SCOPE는 1→2 한 단계만 구현. 이번 기획에선 3단계까지 설계 두고 코드는 점진적.

## 4. 슬롯·직군 매핑

기존 코드의 4 슬롯 구조 유지. `sound` 슬롯은 `qa`로 리네임.

| 슬롯 (코드) | 직군 표시명 | 비어 있을 때 |
|---|---|---|
| `planning` | 기획 / **PM** | 진행도 ↓ (스펙 모호) |
| `graphics` | 디자인 / **UI·UX 디자이너** | Appeal ↓ (시안 부족) |
| `qa` *(기존 sound)* | **QA** | BugDebt 가속 (테스트 늦음) |
| `programming` | 개발 / **개발자** | Progress 정체 |

QA가 1단계에선 비어 있고 2단계에서 합류 — "스타트업 초창기 QA 늦게 합류" 패턴 그대로. 도메인 매칭 로직은 기존 정/오배치 그대로.

## 5. 장르·테마 매핑

### 장르 — 서비스 카테고리

| 코드 | 새 라벨 | 특성 |
|---|---|---|
| G1 | **광고/AdTech** | MVP 짧음, 매출 변동 ↑ (기존 G1 modifier 유지) |
| G2 | **커머스 플랫폼** | 개발 부담 ↑, 평판 롤러코스터 |
| G3 | **AI/추천 서비스** | UI 부담 ↓, 알고리즘 비중 ↑ |

### 테마 — 조직 문화

| 코드 | 라벨 | 특성 |
|---|---|---|
| T1 | 야근과 치킨 | 한국 IT 시그니처 (그대로) |
| T2 | 회의가 레벨이다 | 진행 ↓ / 버그 ↓ |
| T3 | 장애 컨퍼런스콜 *(기존 버그한테 잡아먹힘)* | BugDebt 누적 ↑ |

기존 도메인 modifier는 그대로 유지. 라벨·플레이버 텍스트만 새로.

## 7. K-X 메메 카탈로그 — 풍자 패턴 뱅크

한국 IT 종사자 대부분이 공유하는 메타-농담. 본가 콘텐츠 카피 걱정 없이 마음껏 가져갈 수 있는 산업 공유 자산. **이벤트의 절반 이상을 이 패턴으로 채울 수 있다.**

### "K-X = X를 빠르게 / 형식만"

| K-X | 실체 |
|---|---|
| K-에자일 | 워터폴 빠르게 + 데일리 스탠드업 |
| K-OKR | 분기마다 새로 쓰는 KPI |
| K-스크럼 | 매일 회의 30분 + 스프린트는 분기에 한 번 |
| K-수평조직 | 호칭만 "ㅇㅇ님" / 결정은 위에서 |
| K-DevOps | 개발자가 운영도 함 |
| K-MSA | 모놀리스를 여러 작은 모놀리스로 |
| K-CI/CD | CI는 자동, CD는 사람이 누름 |
| K-자율좌석 | 어차피 자기 자리 똑같음 |
| K-재택근무 | 9시 슬랙 온라인 + 노트북 항상 켜둠 |
| K-1:1 미팅 | 이번주 진척 보고 |
| K-360도 평가 | 누가 누구한테 했는지 다 앎 |
| K-MBO | 목표는 부장이 정해 줌 |
| K-코드리뷰 | "👍" 한 글자 |
| K-TDD | 테스트는 마지막 주에 |
| K-혁신 | 또 조직 개편 |
| K-DX | TF 만들기 |
| K-사일로 해체 | 부서 이름 바꾸기 |
| K-PoC | 임원이 보고 싶어해서 만든 데모 |
| K-온보딩 | 첫날 노트북 + "환영합니다" → 알아서 |

### "X 라고 적혀 있는데 실제로는 Y"

| 적힌 것 | 실제 |
|---|---|
| 자율 | 알아서 야근 |
| 워라밸 | 워크 = 라이프 |
| 강제 아닙니다 | 강제 |
| 캐주얼 데이 | 어차피 매일 |
| 자기계발비 | 사용 시 "어디 가셨어요?" |
| 사이드프로젝트 OK | 본업 끝나고 |
| 휴가 자유 사용 | 부서 분위기 봐 가며 |
| "고견 부탁드립니다" | 답정너 |
| "잠깐 시간 되세요?" | 1시간 회의 |
| "이번 분기는 부채 청산" | 다음 분기에도 부채 청산 선언 |

### 회의·결정의 K 모먼트

- "결정은 미팅 끝나고 따로"
- 임원 PT = 슬라이드 100장
- 그룹콜 = 한 명이 30분 말함
- "확인하셨나요?" 압박
- 새벽 슬랙 = 결정해 놓고 알림
- 단톡·슬랙·이메일에 같은 내용 3번
- "결재 라인 줄이자" → 라인 추가됨

### 일정·릴리스의 K 모먼트

- 프리징 직전 스코프 추가
- 점심·금요일 배포 금기 (어겼다가 장애)
- D-1 일주일 야근
- "이번 분기는 안정화" → 다음 분기에도 안정화
- 장애 컨콜 = 진단 30분 / 책임 30분 / 복구 5분
- 출시 직후 트위터 검색
- 출시 후 대기조

### 도구·기술의 K 모먼트

- 노션 쓰는데 첨부는 카톡
- 슬랙 두고 진짜 일은 카톡
- Figma 라이선스 1개 (돌려 씀)
- 레거시 주석 = "왜 이런지 모름"
- git blame = 본인
- ChatGPT 도입 → "이거 검증된 거?"
- 새 프레임워크 → "왜 굳이?"

### 직급·평가의 K 모먼트

- 호칭만 "ㅇㅇ님" / 결정은 동일
- 분기 평가 = 부장이 점수 매김
- 면접관 차출 = 자기 일도 못 함
- "임원 의견 = 회의 결론"
- 신임 임원 = 모든 거 reset
- 시니어 = 다 함
- 인턴 = 정직원 일

### 9.2 트레이트 카탈로그

특수 트레이트는 일부 직원만 보유. **모든 직원**은 성향 1개(급진/보수)는 반드시 가짐.

| 트레이트 | 효과 |
|---|---|
| 단톡 활성도 높음 | 임원 단톡 이벤트 stamina drain ↑, BD 협상 이벤트 매출 + |
| 회의 좋아함 | 30분→2시간 회의 이벤트 사기 영향 ↓ |
| 완벽주의 | 폴리싱 효율 ×1.2, 연체 시 사기 −2/주 |
| 리팩터 광 | C1 레거시 이벤트에서 BugDebt 감소량 ×1.5 |
| AI 도구 보수적 | C2 보류 시 사기 +5, 도입 시 −5 |
| 테스트 자동화 광 | BugDebt 기본 누적 −1/주 |
| **고인물** ★ | effective skill ×1.3, 단 신기술 도입 이벤트 시 사기 −10 |
| **입 개발 (해커톤 빌런)** ★ | 해커톤·발표 이벤트에서 boost / 평소 progress 기여 ×0.7 |
| **재택 빌런** ★ | 재택 토글 ON 시 5% 확률로 발현; 발현 시 progress 기여 −50% (단순화: 다음 회고 이벤트에서 자동 노출) |

### 9.3 성향 (급진 vs 보수)

모든 직원은 둘 중 하나. 신규 채용 시 무작위 부여(트레이트와 결합).

| 성향 | 효과 |
|---|---|
| **급진파** (신기술 신봉) | C2 AI 도입·C3 새 프레임워크 도입 시 사기 +5; "안정 운영" 선택 시 −5 |
| **보수파** (안정 운영) | C2/C3 도입 시 사기 −5; 야근/긴급 이벤트에서 stamina 영향 −1 |

**J2 (급진파 vs 보수파 충돌)** 이벤트는 양 성향 직원이 동시에 있을 때 트리거.

## 11. 자료 조사 가이드

### 원칙

> 사용 가능한 것은 **트로프(산업 일반 현상)**이고, **특정 작품의 표현은 그 작품의 것**.

휴리스틱: **두 곳 이상에서 발견되는 같은 패턴 = 산업 공유 자산** → 안전하게 트로프 뱅크에 추가.
한 작품에서만 발견되는 표현 → 손대지 않음.

### 자료 카테고리별 활용법

| 유형 | 사용 방식 |
|---|---|
| **1차 소스 (가장 안전)** | 본인 회사 경험, 동료·전 동료 인터뷰. 30분만 친구 둘 만나도 카탈로그 절반 채워짐. 저작권 무관. |
| **공개 통계·기사** | 노동·근무환경 통계, 산업 보고서, 일반 매체 IT 기사. 사실은 저작권 보호 X — 인용·요약 OK. |
| **온라인 커뮤니티** | OKKY·블라인드·디시 IT·잡플래닛 등 — **트렌드 파악용**. 개별 글 옮기기 X, 공통 패턴만 추출. |
| **개발자 매체·뉴스레터** | 회사 문화 다룬 칼럼 — 산업 현상 식별 도움. 인용은 짧게 출처. 카피는 본인이. |
| **웹툰·영상 (영감만)** | 보면서 산업 일반 패턴인 것 vs 작가 창작인 것 구분. 전자만 트로프 뱅크. 후자는 손대지 않음. |
| **컨퍼런스·팟캐스트** | DEVIEW·FE Conf 등 조직 문화 트랙 발표 — 사례 공개. 사실 정보로 활용. |

### 추출 워크플로우

1. 카테고리 잡고 시작 (위 6 카테고리 그대로)
2. 본인 경험 30분 자유 작성
3. 친구 2-3명 짧은 대화 ("최근에 황당했던 모먼트?")
4. 공개 자료 검색 — 산업 일반 현상으로 굳어진 것만
5. 트로프 뱅크에 정리
6. 게임 카피는 본인이 새로 작성

### 안전 라인

- 특정 회사·서비스·인물 이름 직접 사용 X
- 메신저는 "단톡"·"슬랙" 일반어. 카카오톡 brand 직격 X
- 직군 비하·인격 공격 X — 자조와 공감의 결로
- 게임 크레딧에 "각종 종사자분들과의 대화·공개 자료 참고" 명시

## 16. 시스템 카탈로그 — 새 메커닉 디테일

기존 v1 메커닉 위에 PIVOT 단계로 들어오는 시스템들. Tier 표기는 §13 슬라이스와 매핑.

### 16.1 출퇴근 (PIVOT-5, Tier 2)

직원별 통근 시간이 매주 stamina에 영향. 사무실 위치마다 다름.

| 사무실 | 평균 통근 | stamina drain/주 |
|---|---|---|
| 분당 셰어오피스 | 50분 | −2 |
| 판교 임대 | 70분 | −4 |
| 판교 자체 사옥 | 60분 | −3 (셔틀 도입 시 −1) |

**셔틀버스 도입** (영구 토글):
- 분기 비용 −100g
- 모두 stamina drain −2/주, 사기 +3

### 16.2 복장 정책 (PIVOT-5, Tier 2)

회사 차원 토글. 직원 사기·효율에 가벼운 영향.

| 모드 | 효과 |
|---|---|
| 자유 (체크셔츠·후드·반바지) | 모두 사기 +3, 효율 +2% |
| 캐주얼 가이드 | 베이스라인 |
| 정장 | 모두 사기 −3 (외부 미팅엔 OK) |

**후드 팀복 도입**: 1회 −100g, 영구 사기 +5. 신규 입사자 자동 적용. 강한 정체성 모먼트.

K 풍자: "캐주얼 데이 도입" 이벤트 → "어차피 매일 캐주얼이라 변화 없음".

### 16.3 재택근무 (PIVOT-5, Tier 2)

토글 1개. 직원 행복 vs 협업 비용 vs 빌런 리스크.

| 상태 | 효과 |
|---|---|
| 사무실 출근 | 통근 drain 적용 (16.1) |
| **재택 ON** | 통근 drain 0, 모두 사기 +5, BugDebt +2/주, 재택 빌런 발현 확률 5%/주 |

**재택 빌런 단순화**: 발현 즉시 회고 이벤트(K1)에서 자동 노출 — "ㅇㅇ씨가 슬랙 그린닷만 키고…". 발견 후 본인 사기 −10, 회사 BugDebt 누적 -3 (지난 손실 부분 보상).

사무실 단계별 가능 여부:
- 1단계: 풀재택 토글 가능
- 2-3단계: 하이브리드 (재택 시 사기 +3, BugDebt +1)

### 16.4 사내 인테리어 (PIVOT-4, Tier 1)

영구 구매. 사무실 단계마다 잠금 해제.

| 항목 | 비용 | 효과 | 잠금 |
|---|---|---|---|
| 캡슐 커피머신 | -50g | 모두 사기 +5 | 1단계+ (기존 G1) |
| 안마의자 | -80g | 모두 stamina drain −1/주 | 1단계+ |
| 사내 카페 | -200g | 모두 사기 +8, 휴식 시 stamina +1/주 | 2단계+ |
| 게임 라운지 | -150g | 모두 사기 +5, 회의 이벤트 stamina 영향 ↓ | 2단계+ |
| 4K 모니터 풀세팅 | -300g | 모두 효율 +5% | 2단계+ |
| 주차장 | -500g | 직원 출근 stress 완화, 통근 drain −1 | 3단계+ |

각 항목 1회 구매로 영구. 누적되어 후반엔 안락한 사옥.

### 16.5 보상·복지 (PIVOT-5, Tier 2)

분기마다 결정 모먼트 + 영구 복지 메뉴.

**성과급 (분기 결정)**:
| 모드 | 효과 |
|---|---|
| 균등 | -50g/명, 모두 사기 +5 |
| 차등 | -50g 평균, 우수 +15 / 평범 +0 / 부진 -3 |
| 미지급 | -0g, 모두 사기 -8 |

**RSU 도입** (영구, -200g):
- 직원 이직 확률 −50% (스카우트 이벤트)
- 출시 보너스 시 직원 사기 ×1.5

**복지 메뉴** (각 영구 토글):
| 항목 | 비용 | 효과 |
|---|---|---|
| 자기계발비 | -30g/분기 | 모두 stamina +1/주 |
| 헬스장 지원 | -50g/분기 | 모두 stamina +2/주 |
| 사내 식당 | -100g 도입 | 모두 사기 +8 영구 |
| 어린이집 지원 | -200g 도입 | 가족 트레이트 직원 사기 +20 |

### 16.6 마케팅 채널 (PIVOT-6, Tier 2)

기존 홍보 3단계 (없음/소/중) 위에 영구 채널 잠금 해제. 출시 패널에서 채널별 토글.

| 채널 | 1회 잠금 비용 | 출시 시 효과 |
|---|---|---|
| 유튜브 광고 | -150g | 매출 ×1.10, 출시 시 -50g |
| 바이럴 (인플루언서) | -100g | 매출 ×1.08, 운에 따라 −20g~+30g |
| SNS 운영 | -80g | Appeal +3 (출시 시), 점진 효과 |
| 컨퍼런스 발표 | -120g | 명성 +1 (장기 누적 평판), Appeal +5 |

기존 PROMO 3단계와 병행. 채널 잠금은 한 번만, 출시 시마다 활성/비활성 선택.

### 16.7 채용 시스템 (v2)

면접 모드 — v2로 미룸. v1은 자동 채용 유지.

플로우 (v2 설계 기록):
1. 사무실 업그레이드 → "채용 공고" 모달
2. 면접 단계 선택 (코딩 테스트 / 화상 / 라이브 코딩 / 컬처핏)
3. 후보 카드 3장 (트레이트 + 표시 스킬)
4. 1명 선택

면접 강도 ↑ → 표시 vs 실제 차이 ↓ + 시간/골드 비용 ↑.

## 17. LLM 시나리오 (희망사항·v2 검토)

게임 시작 시 사용자가 입력한 키워드/아이디어 → LLM이 시나리오 생성 → 게임 데이터에 주입. v1에는 들어가지 않음. 향후 검토.

세 옵션:

| 옵션 | 작동 방식 | 장단 |
|---|---|---|
| **1. 온라인 LLM** | 사용자 API 키 또는 자체 endpoint → JSON 시나리오 | 무한 다양성. 키 관리·비용·인터넷 의존 |
| **2. 로컬 LLM (브라우저)** | WebLLM / 작은 모델 in-browser | 오프라인 OK. 모바일·저사양 느림. 작은 모델 품질 불안 |
| **3. 프리셋 + 키워드 치환** | 미리 만든 시나리오 50개 + 사용자 키워드로 텍스트 일부 치환 | LLM 없이 가능. 다양성 제한 |

권장 우선순위: 옵션 3을 먼저 → 옵션 1을 "고급 모드"로.

입력 예시:
- 사용자: "AI로 책 요약해 주는 서비스"
- 출력 (JSON):
  ```
  {
    "companyName": "북스마트",
    "firstProject": { "name": "한 줄 요약 봇", "genre": "G3" },
    "events": [{ "title": "출판사 항의", ... }]
  }
  ```

구현 복잡도: 중간. v2 진입 후 메커닉 안정화 시 검토.
