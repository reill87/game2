# 수치 설계 — 경영 시뮬 (v0.1 초안)

> 『게임개발 스토리』식 루프·[PRODUCT_LOOP.md](./PRODUCT_LOOP.md)와 맞춤. **플레이테스트 후 조정 전제.**  
> 예전 RPG용 표는 제거됨 — 필요 시 git 히스토리 참고.

## 1. 스케일·단위

- **BugDebt:** 0~100 정수, **100에서 클램프**.
- **Appeal(해금 후):** 0 이상 누적. 100마다 레벨이 오르는 장기 성장축이며, 리뷰 반영은 soft-cap 곡선으로 완만하게 증가.
- **성공 요소:** 기술/UX/창의/시장 4축, 각 0~100. 개발 주차·AP·이벤트가 올리고 리뷰 점수에 보너스로 반영.
- **Progress:** 0~100%.
- **골드:** v1 튜토리얼 첫 매출 목표 대역 **약 300~500** (이후 작품에서 인플레).
- **초반 완충:** 첫 플레이 시작 자금 **240g**, 첫 3작품 매출 보정 **+120 / +80 / +40g**.

## 2. 플레이 타임 가정

| 구간 | 목표 |
|------|------|
| 첫 작품(튜토리얼) 실플레이 | **8~15분** (배치·8~12주 틱·출시·결과까지) |
| v1 전체(2~3작 + 업그레이드 여지) | **45~90분** |

## 3. 개발 틱 (주 1회) — G1 + T1, 정배치 가정

> “정배치” = 개발자→프로그래밍, 기획자→기획, 디자이너→그래픽, 사운드 빈칸.

| 항목 | 초안값 | 비고 |
|------|--------|------|
| Progress 주당 증가 | **+9~11%** (평균 **+10%**) | 약 **10주**면 100% 근접; 장르 보정으로 ±1주. |
| BugDebt 주당 증가(야근 **OFF**) | **+6** | 야근 ON이면 아래 참조. |
| BugDebt 주당 증가(야근 **ON**) | 기본 **+6**에 **+4** 가산 → **+10** | Progress 보너스와 함께 씀. |

구현상 튜토리얼은 QA/마케팅/데이터 슬롯 부재로 후반 sprint가 늘어지지 않도록 Progress **×1.55**, BugDebt **×0.55** 보정을 적용한다.

2작품부터는 회사가 커질수록 제품 scope도 커진다. `productIndex`, 사옥 단계, 목표 주차, primary+support 배치 인원 6명 초과분에 따라 진행도 기여를 나누는 scope multiplier를 적용한다. 특히 20~30주짜리 후반 프로젝트는 실제 작업량도 같이 커져, 고스펙 팀이어도 5주 만에 끝나는 구조를 피한다.
scope는 보상도 같이 키운다. 큰 프로젝트는 BugDebt 주간 증가를 최대 **÷1.8**까지 완화하고, scope 초과분의 **90%**를 매출 배수로 환산한다. 방향성은 “프로젝트를 오래 운영하고, 출시 때 크게 회수”이다.
리뷰 점수는 BugDebt/연체/폴리싱/홍보 외에 Appeal, 팀 평균 컨디션, 정배치율, 프로젝트 체급(scope), 성공 요소(기술/UX/창의/시장)를 반영한다. Appeal은 무제한 누적되지만 리뷰 점수에는 soft-cap으로 반영해 장기 성장감과 점수 안정성을 같이 유지한다.
각 슬롯은 다른 성공 요소를 올린다. 개발은 기술, 디자인은 UX/창의, 기획은 시장/창의, QA는 기술/UX, 마케팅은 시장, 데이터는 시장/기술에 주로 기여한다. AP 행동과 중간 이벤트도 이 요소를 직접 조정해 “그 주에 무엇을 개선했는지”가 남도록 한다.
primary+support를 모두 채운 뒤 남는 직원은 직접 Progress를 만들지 않지만, 백오피스/랩/리서치로 성공 요소와 BugDebt 억제에 기여한다. 4단계 이상 사옥에서 정원 18~24명까지 늘어도 “12명 이후는 낭비”가 되지 않도록, 최대 12명의 미배치 직원을 보조 기여로 반영한다.

**야근 ON (해당 주):**

- Progress 증가에 **×1.18** (또는 +1.8%p 정도 — 구현 편한 쪽).  
- BugDebt 증가에 위 가산 적용.

*첫 작품에서 야근 토글을 튜토리얼 끝에서만 짧게 소개할지, 아예 2작부터 열지는 UI 정책.*

## 4. 오배치 패널티 (정책 B)

직원 1명이 **자기 직무와 맞지 않는** 슬롯에 있을 때(예: 개발자→기획):

| 효과 | 초안값 |
|------|--------|
| 그 직원이 내는 **진행/품질 기여** | 정배치 대비 **×0.50** |
| 해당 주 **추가 BugDebt** | 직원당 **+2** (같은 주 여러 명 오배치면 합산) |

“반쯤 맞는” 슬롯(예: 기획자→그래픽)은 v2에서 **×0.75** 등으로 세분화 가능. v1은 **맞음 / 안 맞음** 이진만 써도 됨.

## 5. 폴리싱 (출시 패널)

| 동작 | 초안값 |
|------|--------|
| 1주 폴리싱 | BugDebt **−12** (최소 0), 게임 내 **1주 경과** |
| 같은 패널에서 연속 폴리싱 | 동일 수치 반복 적용 가능 |

튜토리얼은 **1주 폴리싱 버튼 1종**만.

## 6. 연체 (목표 주 초과)

목표 주를 넘긴 **매 1주**마다 (개발 틱 또는 폴리싱 주 포함):

| 페널티 | 초안값 |
|--------|--------|
| 골드 | **−8** (0 미만으로는 안 가게 클램프) 또는 **0이면 사기·명성 소량** — v1은 골드만 단순화 권장 |

## 7. 홍보 (두 번째 작품~, 출시 패널)

> v0.2 튜닝 — v0.1(소 ×1.08 / 중 ×1.18)은 score 50~70 구간에서 net 골드가 마이너스라
> "누르면 손해" 함정이었음. 첫 작품 목표 대역(150~400 골드, score ≈ 50)에서 손익이
> 흑자가 되도록 상향. 추후 명성 시스템이 들어오면 다시 보수적으로 내릴 수 있음.

| 단계 | 골드 소비 | 효과 |
|------|-----------|------|
| 없음 | 0 | 배수 **1.00**, 리뷰 +0 |
| 소 | **40** | 매출 배수 **1.18**, 리뷰 **+2** |
| 중 | **100** | 매출 배수 **1.40**, 리뷰 **+5** |

대표 net 손익 (score 66 base, revenue 315 가정):
- 없음 → +315
- 소 (cost 40, ×1.18) → 378 − 40 = **+338** (+23 vs 없음)
- 중 (cost 100, ×1.40) → 459 − 100 = **+359** (+44 vs 없음)

리뷰·매출 **최종 공식**은 구현 시 한 함수로 모으고, 위 값은 **튜닝 노브**로 취급.

## 8. 조정 순서 (권장)

1. 정배치·야근 OFF만으로 **10주 전후에 Progress 100%** 되는지 확인.  
2. BugDebt가 출시 시점에 **30~70** 대에 자주 오는지 (너무 낮으면 출시 결정이 무미건조).  
3. 오배치 런에서 **3~4주 더 걸리거나** BugDebt **80+** 로 가는지.  
4. 폴리싱 1회가 체감될 만큼 BugDebt를 낮추는지 (**−12** 조정).
