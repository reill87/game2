# 작품 루프 (게임개발 스토리식) — 초안

> 경영 시뮬 핵심. RPG 전투 없음. 카피 가드레일은 [CONTENT_POLICY.md](./CONTENT_POLICY.md).

## 한 작품 사이클 — 단계

1. **기획** — 장르·테마(·v2에서 플랫폼/타깃) 선택  
2. **배치** — 직원을 기획/그래픽/사운드/프로그래밍 등 담당에 배치  
3. **개발 진행** — 틱마다 진행도·버그·품질 누적, 교육·압박·이벤트  
4. **마감** — 출시일 vs 폴리싱·버그 수정 자원 배분  
5. **출시·홍보** — v1에서는 단순화 가능 (마케팅 슬라이더 1개 등)  
6. **결과** — 리뷰·매출·명성 → 자금·연구·**사무실 업그레이드**로 연결 (타이쿤 축)

## 장르·테마 — v1 최소 세트 (3×3)

공통 밸런스: **장르 베이스 보정 + 테마 보정**만으로 9조합 운용 가능. 조합별 **리뷰 첫 줄·이벤트 카피**만 차별화해도 됨.

### 장르 (가칭)

| ID | 이름 | 요약 |
|----|------|------|
| G1 | 초단타 터치 | 기간 짧음, 완성도 낮아도 매출 변동 큼 |
| G2 | 한 판마다 새로 | 개발·디버깅 부담↑, 명성 롤러코스터 |
| G3 | 말이 주인공 | 그래픽 부담↓, 기획/시나리오 비중↑ |

### 테마 (가칭)

| ID | 이름 | 요약 |
|----|------|------|
| T1 | 야근과 치킨 | 일정 압박 이벤트와 잘 맞는 기본 테마 |
| T2 | 회의가 레벨이다 | 진행 느림·조직 메타 보정 여지 |
| T3 | 버그한테 잡아먹힘 | 버그 포인트 기본↑, 디버깅 시너지 |

## 튜토리얼 첫 작품

- **고정 조합:** **G1 + T1** (초단타 터치 × 야근과 치킨)  
- **의도:** 짧은 사이클로 매출·리뷰·자금 흐름을 한 번에 이해시키기  
- UI에서 3×3을 보여 주더라도 v1은 **선택 없이 이 조합으로 시작** (옵션 A 확정)

## 개발 진행 — 틱 모델 (v1 초안)

### 시간 단위

- **1틱 = 1주** (게임 내 시간). 원작과 비슷하게 “몇 주 짜리 프로젝트”로 이해하기 쉬움.
- **프로젝트 길이(목표 주 수)** 는 장르·규모에서 오는 **기본값** + 이벤트 보정.  
  - **G1 튜토리얼** 예: **8~12주** (짧게 끝나 첫 매출까지 도달 빠르게).

### 플레이어에게 보이는 칸 (v1 확정: 점진 해금)

| 칸 | 범위 | 의미 |
|----|------|------|
| **진행도 (Progress)** | 0~100% | 기능 구현·콘텐츠 채움. **100%면 출시 가능 상태** (버그 많으면 페널티). |
| **버그 부채 (BugDebt)** | 0~100 | 높을수록 출시 후 리뷰·매출 리스크, 마감 이벤트에서 “야근 디버깅” 카피와 연동. |
| **완성 매력 (Appeal)** | 0~100 | 그래픽·사운드·폴리싱을 한 덩어리로 본 **기대 품질**. 리뷰 기대치에 직접. 원작 4분할(그래픽/사운드/버그/재미)은 **v2** 후보. |

**UI 노출 순서 (옵션 C + 해금 타이밍 B 확정):**

1. **첫 작품 전체(기획~출시):** **Progress + BugDebt** 두 칸만 표시. Appeal은 **내부적으로 0 고정 또는 미계산** — 온보딩은 “일정·버그”에만 집중.  
2. **두 번째 작품 기획 화면 진입 시:** **Appeal 칸 해금** — 장르·테마 선택 전후에 짧은 팝업/대사로 “이제는 완성 매력도 신경 써야 한다”고 연결.

백엔드 수식은 구현 편의에 따라, 첫 작품 동안 Appeal을 끄었다가 **두 번째 작품부터** 틱 연산에 넣어도 됨.

### 매 틱 갱신(개념식)

직원 **i**가 담당 **b**(기획/아트/사운드/프로그래밍)에 배치되어 있을 때:

- **진행 기여** ∝ `매칭 계수(직원 직무 × 담당)` × `스킬` × `(1 + 교육 보너스)` × `장르·테마 계수`
- **버그 증가** ∝ `진행 기여의 일부` × `야근/압박 플래그` × `G2·T3 등 보정` − `디버깅에 투입된 인력`
- **Appeal 증가** ∝ 아트/사운드/기획 비중에 분배 (프로그래밍은 Progress에는 크게, Appeal에는 덜)

*정확한 계수는 스프레드시트·플레이테스트에서 조정.*

### 압박(야근) — 풍자 연출용 토글

- **켜면:** 해당 주 `Progress` 보너스, `BugDebt` 페널티.  
- **끄면:** 안정적이지만 일정이 늘어날 수 있음.  
- 카피는 [CONTENT_POLICY.md](./CONTENT_POLICY.md) 범위 안에서만.

### 마감과의 연결

- **진행도 100%** 도달 시: “출시 준비” 단계로 — 버그·매력에 따라 **추가 폴리싱 주**를 줄지, 바로 낼지 선택 (v1은 단순 분기).  
- **목표 주 수 초과:** 연체 페널티(외주비·사기 등 **가벼운 게임 내 자원**)만 사용. 실직·질병 소재 금지.

## 배치 튜토리얼 (첫 작품 전용)

### 담당 슬롯 (원작 4분할과 동일 이름)

| 슬롯 | 설명 |
|------|------|
| **기획** | 스펙·콘텐츠 양, Progress·(해금 후 Appeal) 기여 |
| **그래픽** | 비주얼, Appeal 비중 큼 |
| **사운드** | 사운드, Appeal·분위기 |
| **프로그래밍** | 구현, Progress 비중 큼, BugDebt와 밀접 |

### 튜토리얼 전제 인원 (1단계 사무실과 맞춤)

- **직원 3명:** 기획자 / 디자이너 / 개발자 (이름은 작품 세계에 맞게 가칭).  
- **슬롯 4개** 중 **사운드는 비움** — G1 초단타에서도 자연스럽고, “구멍 나 있는 팀”을 한 번 보여 줌.  
- 카피 예: *“사운드는… 이번엔 효과음 사이트로 버틴다고 했죠.”* ([CONTENT_POLICY.md](./CONTENT_POLICY.md) 범위 유지)

### 단계별 비트 (권장 순서)

1. **배치 화면 진입** — 4칸·3명이 보임. 상단에 “담당에 맞게 올려 두면 매주 기여합니다” 한 줄.  
2. **스텝 ① (강조)** — **프로그래밍** 칸 하이라이트. *“개발자는 여기. 안 그러면 진행도가 굳는다.”*  
3. **스텝 ②** — **기획** 칸 강조. 기획자 배치.  
4. **스텝 ③** — **그래픽** 칸 강조. 디자이너 배치.  
5. **스텝 ④** — **사운드** 빈 칸 설명만. *“비어 있으면 그만큼 느리거나 매력이 덜 붙을 수 있음”* — 수치는 G1에서 완만하게.  
6. **확인** — `[개발 시작]` 또는 `[다음 주로]` 활성화. **오배치 정책 B 확정:** 잘못 올려도 **진행은 항상 가능**. 대신 **기여 감소·BugDebt 증가 등 패널티**로 “담당이 왜 있는지”를 수치로 체감. 짧은 힌트 카피는 가능 (*“…여기 넣는 게 맞았을 텐데요?”*).  
7. **첫 틱 진입** — “이제 한 주가 갔습니다” + Progress/BugDebt 변화 요약.

### 입력 방식 (구현 무관)

- 드래그 앤 드롭 / 리스트에서 슬롯 탭 / 카드 클릭 후 슬롯 클릭 중 프로젝트에 맞게 하나만 지원하면 됨.

## 마감·출시 UI (v1 초안)

### 진입 조건

- **진행도 Progress = 100%** 가 되는 순간 **「출시 준비」** 상태로 전환.  
- 개발 화면 상단에 배너 또는 **모달 자동 표시** (플레이어가 놓치지 않게).  
- **v1:** Progress **100% 미만이면 출시 불가** (얼리액세스·미완 출시는 v2).  
- **목표 주차를 넘긴 경우(연체):** 같은 패널 상단에 **연체 경고 배너** — 외주비·사기 등 **가벼운 자원 페널티** 요약만 (정책 문서 준수).

### 화면 구역 (세로 UI 기준)

1. **헤더** — 작품명(가칭), 장르·테마 뱃지(G1+T1 등), **현재 주 / 목표 주**.  
2. **상태 요약** — Progress는 **완료 체크**로만 표시. **BugDebt** 게이지 강조.  
   - **Appeal** 은 **두 번째 작품부터** 이 화면에도 표시(해금 정책과 동일). 첫 작품은 Appeal 영역 없음.  
3. **출시 전 선택**  
   - **주 버튼:** `[지금 출시하기]`  
   - **폴리싱:** `[N주 더 다듬기]` — v1에서는 **N = 1 또는 2~3 고정 스텝** 중 하나만 지원해도 됨.  
     - 한 번 탭할 때마다 **게임 내 1주 경과**, BugDebt **고정량 감소**(수치는 밸런스), 목표 주 초과 시 연체 페널티 누적 가능.  
     - 폴리싱 중에도 **배치 화면으로 돌아가지 않음** — “출시 직전 버그만 잡는 주”로 간주해 UI는 이 패널에 머무르거나, 동일 레이아웃에서 “이번 주는 폴리싱” 한 줄만 바뀜.  
   - **툴팁/경고:** BugDebt 구간별 문구 (예: 70↑ *“이대로 내면 리뷰가 험할 수 있습니다.”* — 과장·풍자).  
4. **홍보 (v1 단순)** — 같은 화면 하단에 두기. (첫 작품 튜토리얼에서는 **숨김** — 위 튜토리얼 절 참고.)  
   - 예: **3단계** `없음 / 소 / 중` 또는 슬라이더 1개 — **보유 골드**에서 차감, 출시 직후 매출·리뷰 분산에만 반영 (세부는 밸런스).  
5. **100% 이후 네비게이션 (A 확정)** — Progress **100% 달성 후**에는 **출시 준비 UI에만 머무름**. 일반 개발 틱으로 돌아가 **진행도를 더 올리는** 동선은 **없음**. 허브로 빠지는 `[닫기]`도 **두지 않음** — **출시 확정** 또는 **폴리싱(1주)** 만 선택하게 해 실수·분기를 줄임. **설정/일시정지**만 예외 허용 가능.

### 출시 확정 후

- 짧은 **전환 연출** (스토어 심사 풍자 카피 1~2초).  
- **결과 화면**으로: 리뷰 헤드라인, 매출, 명성 변화, **자금 → 사무실 업그레이드** 힌트.

### 튜토리얼 첫 작품 (UI 단순화 — **B 확정**)

- **`[지금 출시하기]`** + **`[1주 더 다듬기]`** 만 노출.  
- **홍보** 컨트롤은 **숨김** — 두 번째 작품부터 같은 패널에 단계형 홍보 도입.  
- 폴리싱 버튼은 **1주 고정** (스텝퍼 복잡도 없음).
